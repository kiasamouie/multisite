Code Organization & Architecture Guidelines

I'm building a modular, component-driven Next.js app. Follow these patterns:

Folder Structure:

/components/ organized by feature/context, not by type
/components/admin/ — admin UI components only
/components/site/ — public-facing site components (nav, footer)
Each feature gets its own folder: /components/feature-name/index.tsx
Sub-components stay near their parent: /components/Resource/SidePanel.tsx
Component Philosophy:

Use Refine.js for data/state logic (useTable, useCreate, etc.)
Write our own lean UI layer in React + Tailwind (no bulky UI libraries like Material-UI)
Favor composition: pass props for config, avoid hardcoding behavior
One concern per file: separate data fetching, rendering, styling
API Layer:

Admin pages use /api/admin/* endpoints (service role, bypasses RLS)
Tenant pages use RLS-enforced endpoints via browser client
Never call Supabase directly from components—use API routes for business logic
Reusability:

Extract to ui or lib only if shared across multiple apps
Keep app-specific logic in src
Pass configuration via props, never hardcode tenant-specific logic
File Naming:

Components: PascalCase (SiteNav.tsx, TenantFlagsView.tsx)
Utilities: camelCase (api-auth.ts, tenant.ts)
Keep imports clean: use barrel exports (index.ts) for folders
Testing Before Shipping:

Run pnpm tsc --noEmit to verify TypeScript
Run pnpm build to verify production build
Always check imports are updated when moving files
